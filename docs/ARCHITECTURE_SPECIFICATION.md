# Architecture Specification: Discover-First MCP Server

> **To build a "discover‑first, assume‑nothing" MCP server** that feels friction‑less for both LLM copilots *and* human developers, this specification translates our discovery-first philosophy into concrete, **Type‑Safe TypeScript** artifacts.

## Executive Overview

We host a **Fastify‑based HTTP service** that mounts MCP **tools** as first‑class endpoints generated by the **official `@mcp/sdk`**.

Every tool is a **pure‑function façade** around New Relic backends, returns a **uniform envelope**, and implements the **3‑step lifecycle**: `discover ➜ analyse ➜ action`.

Type safety, validation, rate‑limits and logging are handled by **composable Fastify plugins**, while **Zod schemas** feed both compile‑time types **and** run‑time JSON‑Schema that the MCP "meta" tools expose for introspection.

## 3. Core Abstractions

### 3.1 Uniform Envelope

```typescript
export interface ResponseEnvelope<T = unknown> {
  ok: boolean;
  data?: T;
  error?: { code: ErrorCode; message: string };
  cursor?: string;
  discoverToken?: string;
}
```

**Key Properties:**
- All Fastify routes return this envelope
- `discoverToken` is a **JWT‑style** string (HS256, 15 min TTL) signed by `plugins/tokens.ts`
- Mutating tools must receive a *fresh* token **and** a single‑use `confirmToken` issued by a prior dry‑run

### 3.2 URN Grammar

```
urn:<domain>:<type>:<id>
```

**Examples:**
- `urn:apm:entity:0b27c1…`
- `urn:metric:infra:0b27c1:system.cpu.percent`
- `urn:alert:policy:abc123`

A Zod `regex` guard ensures strings match this grammar at the API edge.

## 4. Tool Interface Contracts

The MCP SDK ships a generic helper:

```typescript
import { defineTool } from '@mcp/sdk/server';

export function makeTool<I, O>(
  name: ToolName,
  input: z.ZodType<I>,
  handler: (ctx: ToolContext, input: I) => Promise<O>,
  options?: ToolOptions,
) {
  return defineTool({ name, inputSchema: input, handler, ...options });
}
```

**Key Features:**
- `ToolContext` injects **logger**, **nrClient**, **redis**, the **Fastify request ID**, and helper to *renew* discover tokens
- The return type is automatically wrapped into `ResponseEnvelope<O>`
- All schemas are Zod-based for compile-time safety and runtime validation

## 5. Concrete Tool Definitions

| Tool | Input Schema Excerpt | Behavior |
|------|---------------------|----------|
| `meta.tool_list` | `{}` | Enumerates all registered tools (reads from SDK registry) |
| `meta.tool_schema` | `{ name: z.string() }` | Returns input+output JSON‑Schema + examples |
| `discover.entities` | `{ domain?: z.string(), name?: z.string(), ... }` | GraphQL `entitySearch` proxy; embeds `discoverToken` |
| `discover.metrics` | `{ entityUrn: z.string(), since?: z.string() }` | Lists metric names only, no datapoints |
| `analysis.query` | `{ nrql: z.string(), accountId: z.number(), timeoutMs?: z.number() }` | Validates with internal NRQL parser; guards LIMIT 100 |
| `action.alert.condition.nrql.create` | `{ policyUrn, nrql, threshold, evalOffset, dry_run, confirmToken? }` | Refuses real mutation unless `dry_run:false` *and* valid `confirmToken` |

**Destructive Action Pattern:**
All `action.*` tools have the decorator:

```typescript
destructiveHint: true,
confirmationRequired: true,
```

This lets an LLM copilot prompt the user automatically per MCP specification.

## 6. Fastify Server Bootstrap

```typescript
import Fastify from 'fastify';
import { mountAllTools } from '@mcp/sdk/server';
import { loggerPlugin } from './plugins/logging';
import { rateLimitPlugin } from './plugins/rateLimit';

export async function buildServer() {
  const app = Fastify({ logger: false });   // pino handled in plugin
  await app.register(loggerPlugin);
  await app.register(rateLimitPlugin, {
    max: 120,
    timeWindow: '1 minute',
  });

  // MCP tools are auto‑mounted under /tools/<name>
  mountAllTools(app);

  return app;
}
```

Fastify's performance advantage in async throughput is well documented.

## 7. Cross-Cutting Plugins

### 7.1 Logging

```typescript
import fp from 'fastify-plugin';
import pino from 'pino';

export const loggerPlugin = fp(async (app) => {
  const logger = pino({ level: process.env.LOG_LEVEL ?? 'info' });
  app.decorate('log', logger);
  app.addHook('onRequest', (req, _res, done) => {
    req.log = logger.child({ reqId: req.id });
    done();
  });
});
```

### 7.2 Rate Limiting

```typescript
import rateLimit from '@fastify/rate-limit';
export const rateLimitPlugin = rateLimit;
```

Supports IP or custom key (e.g. API key) + Redis backend for cluster mode.

## 8. Validation & Security

- **Input Validation**: All inbound payloads pass through `inputSchema.safeParse`; violations return `400` with `error.code = "VALIDATION_ERROR"`
- **JWT Security**: JWT secrets live in `CONFIG.JWT_SECRET`; rotated via SSM Parameter Store
- **Security Headers**: Fastify's built‑in Helmet plugin enables HSTS/CSP etc.
- **Audit Logging**: Every `action.*` call streams to New Relic Logs for compliance

## 10. Observability Inside the Observability Server

- **Custom Events**: Each tool invocation records a `McpToolInvocation` custom event with `{tool, ok, latencyMs}`
- **SLO Monitoring**: P95 latency SLO alarms use NRQL alerts; values surfaced through `analysis.query` itself
- **Distributed Tracing**: OpenTelemetry traces exported via `@fastify/otel` plugin, stitching requestID ↔ traceID

**Meta-Observability**: Users can **observe the observer** through the same MCP tools.

## 11. Discover-First Lifecycle

### Phase 1: DISCOVER
```typescript
// Never assume data structure exists
const entities = await discover.entities({ 
  domain: 'APM', 
  name: '*web*' 
});

const metrics = await discover.metrics({ 
  entityUrn: entities.data[0].urn 
});
```

### Phase 2: ANALYSE  
```typescript
// Build informed queries based on discovered structure
const nrql = buildInformedQuery(entities.data, metrics.data);
const analysis = await analysis.query({ 
  nrql, 
  accountId,
  timeoutMs: 10000 
});
```

### Phase 3: ACTION
```typescript
// Always dry-run first, then confirm
const dryRun = await action.alert.create({
  policyUrn: 'urn:alert:policy:web-alerts',
  nrql,
  threshold: calculateThreshold(analysis.data),
  dry_run: true
});

// User reviews and confirms
const result = await action.alert.create({
  ...dryRun.data,
  dry_run: false,
  confirmToken: dryRun.confirmToken
});
```

## 12. Future Extensions

1. **Long‑Running Jobs** → return `jobUrn` and stream via MCP *chunked* transport
2. **Multi‑Tenant** → prefix Redis keys by `accountId` & sign tokens with per‑tenant secret
3. **GraphQL Transport** → the SDK also exposes `createGraphQLServer()` for real‑time subscriptions
4. **Edge Runtime** → Fastify ≥ 5 experimental worker mode allows Cloudflare Workers deployment
